# Agent 构建器

为任何领域构建AI Agent——客户服务、研究、运营、创意工作或专门的业务流程。

## 核心哲学

> **模型早已懂得如何成为Agent。你的工作是让它能够行动。**

Agent不是复杂的工程。它只是一个简单的循环，邀请模型去行动：

```
循环：
  模型看到：上下文 + 可用能力
  模型决定：行动或回应
  如果行动：执行能力，添加结果，继续
  如果回应：返回给用户
```

**就这么简单。** 魔法不在代码中——它在模型中。你的代码只是提供了机会。

## 三个要素

### 1. 能力（它能做什么？）

Agent可以执行的原子操作：搜索、读取、创建、发送、查询、修改。

**设计原则**：从3-5个能力开始。只有当Agent反复因为缺少某个能力而失败时，才添加更多。

### 2. 知识（它知道什么？）

按需注入的领域专业知识：策略、工作流、最佳实践、模式。

**设计原则**：让知识可用但不强制。在相关时加载，而不是预先加载。

### 3. 上下文（发生了什么？）

对话历史——将行动连接成连贯行为的线索。

**设计原则**：上下文是珍贵的。隔离产生噪音的子任务。截断冗长的输出。保护清晰性。

## Agent设计思维

在构建之前，理解：

- **目的**：这个Agent应该完成什么？
- **领域**：它在什么世界运作？（客户服务、研究、运营、创意...）
- **能力**：哪3-5个行动是核心？
- **知识**：它需要访问什么专业知识？
- **信任**：你可以把什么决策委托给模型？

**关键**：信任模型。不要过度工程。不要预先指定工作流。给它能力，让它推理。

## 渐进式复杂性

从简单开始。只有当实际使用揭示需求时才增加复杂性：

| 级别 | 添加什么 | 何时添加 |
|------|----------|----------|
| 基础 | 3-5个能力 | 总是从这里开始 |
| 规划 | 进度跟踪 | 多步骤任务失去连贯性时 |
| 子Agent | 隔离的子Agent | 探索污染上下文时 |
| 技能 | 按需知识 | 需要领域专业知识时 |

**大多数Agent永远不需要超过第2级。**

## 领域示例

**业务**：CRM查询、邮件、日程、审批
**研究**：数据库搜索、文档分析、引用
**运营**：监控、工单、通知、升级
**创意**：资产生成、编辑、协作、审查

模式是通用的。只有能力会变化。

## 关键原则

1. **模型就是Agent**——代码只是运行循环
2. **能力赋能**——它能做什么
3. **知识指导**——它知道如何做什么
4. **约束聚焦**——限制创造清晰性
5. **信任解放**——让模型推理
6. **迭代揭示**——从最小开始，从使用中演进

## 反模式

| 模式 | 问题 | 解决方案 |
|------|------|----------|
| 过度工程 | 需求之前的复杂性 | 从简单开始 |
| 太多能力 | 模型困惑 | 开始时3-5个 |
| 僵化工作流 | 无法适应 | 让模型决定 |
| 预加载知识 | 上下文膨胀 | 按需加载 |
| 微管理 | 削弱智能 | 信任模型 |

## Agent思维方式

**从**："我如何让系统做X？"
**到**："我如何让模型能够做X？"

**从**："这个任务的工作流是什么？"
**到**："什么能力有助于完成这个任务？"

最好的Agent代码几乎是无聊的。简单的循环。清晰的能力。干净的上下文。魔法不在代码中。

**给模型能力和知识。信任它自己弄清楚其余的。**
